# 洞察文档修正

## 定位

这是 codebase-overview、codebase-deep-dive、project-evaluation 三个技能的**配套修正技能**。

三个生成技能负责"从零创建文档"，本技能负责"在用户提问和讨论中持续完善文档"。

## 核心规则

当用户针对 `.code-insight/` 或 `.project-eval/` 中的内容提问时，你必须**同时做两件事**：

1. **回答问题**
2. **将回答写回对应的 md 文件**

不存在"只口头回答不更新文件"的情况。回答和回写是一个原子操作。

## 工作流程

### 第一步：定位目标文件

根据用户提问的内容，判断涉及哪个文件：

- 提问关于项目全景、模块概况、架构图 → `.code-insight/main.md`
- 提问关于某个具体模块 → `.code-insight/[模块名]/main.md` 或 `.code-insight/[模块名].md`
- 提问关于项目评估、竞品、评分 → `.project-eval/main.md`
- 提问关于配置项 → `.project-eval/config-guide.md`

如果不确定涉及哪个文件，先用 Glob 搜索 `.code-insight/` 和 `.project-eval/` 目录结构，找到最相关的文件。

### 第二步：读取文件上下文

用 Read 工具读取目标文件中与问题相关的段落，理解当前文档中是怎么写的。

### 第三步：回答 + 回写

回答用户的问题，然后立即用 Edit 工具更新文件。回写方式取决于场景：

| 场景                             | 回写方式                                     |
| -------------------------------- | -------------------------------------------- |
| 用户问"XX 是什么意思"            | 在原段落中扩展解释，或在该段落后新增一个小节 |
| 用户说"这里不对"                 | 直接修正文件中的错误内容                     |
| 用户追问导致发现新的关联         | 在最相关的位置补充说明                       |
| 用户的问题揭示了之前表述不够清晰 | 改写原段落使其更易理解                       |
| 涉及跨文件修正                   | 所有相关文件都要更新                         |

### 第四步：告知用户

回答末尾用一句话说明更新了哪个文件，例如："已更新到 `.code-insight/ai/main.md`"。

## 判断标准

### 必须回写的情况

- 用户问"XX 是什么意思"、"这里不太对"、"为什么是这样"
- 用户说"XX 和 YY 的关系是什么"
- 用户纠正了文档中的错误
- 用户追问后你发现之前的表述不够准确
- 用户要求补充某个方面的说明

### 不需要回写的情况

- 用户明确说"不用写回去"或"只是随便问问"
- 纯操作性问题（如"这个文件在哪"、"怎么打开"）
- 与文档内容完全无关的对话

### 如何处理深入研究

如果用户的问题需要你**阅读源码**才能回答（比如"这个函数到底是怎么工作的"），完整流程是：

1. 读取源码，理解逻辑
2. 回答用户
3. 把你从源码中了解到的关键信息写回 md 文件——**不是把源码贴进去**，而是把"这段代码意味着什么"的解释写进去

## 回写原则

- **增量编辑，不要重写整个文件**。使用 Edit 工具修改特定段落，不要用 Write 工具覆盖全文
- **保持文档结构**。新增内容应该融入已有的章节结构，不要在文末堆积"补充说明"
- **与已有内容风格一致**。先读取文件，理解文档的语气和深度，保持一致
- **如果修正影响多个文件，全部更新**。比如 module 文档发现 main.md 的模块描述有误，两个都要改
