# 代码库全景分析

为整个代码库生成结构化、图文并茂、由浅入深的全景理解文档。产出物是项目根目录下的 `.code-insight/` 目录，其中 `main.md` 是所有后续探索的唯一入口。

## 适用场景

- 用户打开或下载了一个新项目，想要快速理解
- 用户问"这个项目是干什么的"或类似的定位性问题
- 用户想要在深入细节之前先建立全局认知

---

## 第一阶段：全量扫描

系统性地阅读整个项目，不要跳过文件或仅凭文件名猜测。

**扫描顺序：**

1. **入口文件优先** — `package.json`、`Cargo.toml`、`pyproject.toml`、`go.mod`、`Makefile`、`docker-compose.yml`、`README.md` 等。这些文件揭示项目身份、依赖关系和构建/运行方式。
2. **配置文件** — `.env.example`、配置目录、CI/CD 文件。这些揭示部署拓扑和环境假设。
3. **源代码** — 遍历完整目录树，阅读每个源文件。对于大型项目（500+ 文件），按优先级阅读：入口点 → 核心逻辑 → 工具函数 → 测试。
4. **测试代码** — 测试就是文档。测试揭示预期行为、边界情况和开发者的思维模型。

---

## 第二阶段：提炼理解

阅读完毕后，围绕以下问题构建内部心智模型（这些问题驱动文档生成，不要作为原始列表输出给用户）：

**问题域：**

- 这个项目解决什么现实问题？
- 目标用户是谁？他们的痛点是什么？
- 如果这个项目不存在会怎样？（这能帮助理清项目的价值所在）

**技术方案：**

- 核心技术路径是什么？（例如："基于事件驱动的管道，监听链上交易并触发通知"）
- 关键技术选型是什么，为什么这样选？（不要只罗列技术栈，要解释背后的逻辑）
- 数据是如何在系统中流动的？信息如何进入、转换、输出？

**架构：**

- 模块之间的边界在哪里？
- 模块之间如何通信？（HTTP、消息队列、共享状态、函数调用等）
- 外部依赖和集成点有哪些？

**难点与风险：**

- 这个项目解决的最难的技术问题是什么？
- 代码中最复杂或最脆弱的部分在哪里？
- 做了哪些技术权衡，原因是什么？

**模块识别 — 将代码库拆分为 3–8 个逻辑模块：**

- 每个模块应当**内聚**（模块内文件紧密相关）、**松耦合**（通过清晰接口交互）、**以功能命名**（如"交易引擎"、"通知系统"、"用户认证"，而不是"src/lib"这种目录名）
- 对每个模块记录：用途（一句话）、关键文件、复杂度估计（简单 / 中等 / 复杂）

---

## 第三阶段：生成 `.code-insight/main.md`

在项目根目录创建 `.code-insight/` 目录，按以下模板生成 `main.md`：

```markdown
# [项目名称] — 代码洞察

> 生成时间: [时间戳]
> 项目路径: [路径]

## 一句话概括

[用一句简洁的话说清楚这个项目是什么、做什么的]

## 项目解决的问题

### 痛点

[描述目标用户面临的真实痛点 — 这是业务/场景层面的描述，不是技术细节]

### 解决思路

[这个项目用什么思路来解决上述痛点？核心创新点或技术选型逻辑是什么？]

## 技术方案

### 整体架构

[用文字简要描述整体架构，然后紧跟一个 mermaid 架构图]

` ` `mermaid
graph TB
    subgraph 用户层
        A["用户请求"]
    end
    subgraph 服务层
        B["API 网关"]
        C["核心服务"]
    end
    subgraph 数据层
        D[("数据库")]
    end
    A --> B --> C --> D
` ` `

### 技术栈

[不是简单罗列，而是说明每个技术选型的理由]

| 技术 | 用途 | 选型理由 |
| ---- | ---- | -------- |
| ...  | ...  | ...      |

### 数据流

[描述数据如何在系统中流动，配合 mermaid 时序图]

` ` `mermaid
sequenceDiagram
    participant U as 用户
    participant S as 服务
    participant D as 数据库
    U->>S: 请求
    S->>D: 查询
    D-->>S: 返回
    S-->>U: 响应
` ` `

## 实现难点与解决路径

[列出 2–5 个关键技术难点，每个难点包含以下内容：]

### 难点 1: [名称]

- **问题**: [这个难点的本质是什么]
- **难在哪**: [为什么这个问题不好解决]
- **解决路径**: [项目采用了什么方案来解决]
- **代码位置**: [关键实现在哪些文件中]

## 项目结构

` ` `[带注释的项目目录树，标注关键目录和文件的用途]` ` `

## 模块地图

> 研究进度: [0/N] 模块已完成
>
> 选择一个模块开始深入研究。完成后在对应行标记 ✅

| #   | 模块     | 说明         | 复杂度 | 详情     | 状态 |
| --- | -------- | ------------ | ------ | -------- | ---- |
| 1   | [模块名] | [一句话说明] | ⭐     | [待生成] | ⬜   |
| 2   | [模块名] | [一句话说明] | ⭐⭐   | [待生成] | ⬜   |
| 3   | [模块名] | [一句话说明] | ⭐⭐⭐ | [待生成] | ⬜   |

## 模块依赖关系

` ` `mermaid
graph LR
    A["模块1"] --> B["模块2"]
    A --> C["模块3"]
    B --> D["模块4"]
` ` `

## 快速上手

### 如何运行

[安装依赖、配置环境、启动项目的具体命令]

### 如何测试

[运行测试的命令和说明]

### 关键配置

[需要关注的配置项及其含义]
```

**重要提示：** 以上模板中的 mermaid 代码围栏使用了 ` ` `（带空格）来转义。生成实际文档时必须使用标准的三个连续反引号 ``` ，否则 mermaid 无法渲染。

---

## 写作规范

### 语言

- 全程使用中文撰写
- 使用精确的技术术语，但在首次出现时给出解释
- 每句话都应承载信息，不写废话

### Mermaid 图表 — 必选项，不是可选项

每份 `main.md` 至少包含以下三种图表：

1. **架构总览图**（graph TB/LR）
2. **数据流图**（sequenceDiagram 或 flowchart）
3. **模块依赖图**（graph LR）

Mermaid 语法注意事项：

- 包含中文、空格或特殊字符的标签必须加引号：`A["标签内容"]`
- `subgraph` 嵌套不超过 2 层
- 每个 `subgraph` 必须有对应的 `end`
- 用 `-->` 表示连线，不要用 `---`
- 写完后在脑中验证语法是否正确再输出

### 深度校准

- 小型项目（<20 个文件）：尽量覆盖每个文件
- 中型项目（20–100 个文件）：覆盖所有模块，重点标注关键文件
- 大型项目（100+ 个文件）：聚焦架构和关键流程，以文件路径引用为主

### 诚实原则

- 如果某段代码令人困惑或文档缺失，直接说明
- 如果项目存在明显问题（安全、性能、架构），在"⚠️ 值得注意的问题"章节中指出
- 如果不确定某个模块的用途，使用"根据代码结构推测，这个模块的作用可能是..."而不是当作事实陈述

---

## 第四阶段：展示与引导

生成 `main.md` 后：

1. 告诉用户这个项目做什么（一句话概括）
2. 识别出了多少个模块
3. 建议从哪个模块开始研究（通常是核心业务逻辑模块，而非基础设施模块）
4. 引导用户："想深入了解哪个模块？"— 这将触发 `module-deep-dive` 技能
