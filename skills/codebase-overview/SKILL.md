# 代码库全景分析

为整个代码库生成结构化、图文并茂、由浅入深的全景理解文档。产出物放到项目根目录的 `.code-insight/` 目录下：

- `main.md` — 当前状态的全景切片（架构、技术栈、模块地图），是所有后续探索的入口
- `evolution.md` — 项目从第一次提交到现在的演进考古，记录作者的思维和技术成长轨迹

## 适用场景

- 用户打开或下载了一个新项目，想要快速理解
- 用户问"这个项目是干什么的"或类似的定位性问题
- 用户想要在深入细节之前先建立全局认知

---

## 第一阶段：全量扫描

系统性地阅读整个项目，不要跳过文件或仅凭文件名猜测。

**扫描顺序：**

1. **入口文件优先** — `package.json`、`Cargo.toml`、`pyproject.toml`、`go.mod`、`Makefile`、`docker-compose.yml`、`README.md` 等。这些文件揭示项目身份、依赖关系和构建/运行方式。
2. **配置文件** — `.env.example`、配置目录、CI/CD 文件。这些揭示部署拓扑和环境假设。
3. **源代码** — 遍历完整目录树，阅读每个源文件。对于大型项目（500+ 文件），按优先级阅读：入口点 → 核心逻辑 → 工具函数 → 测试。
4. **测试代码** — 测试就是文档。测试揭示预期行为、边界情况和开发者的思维模型。

---

## 第二阶段：提炼理解

阅读完毕后，围绕以下问题构建内部心智模型（这些问题驱动文档生成，不要作为原始列表输出给用户）：

**问题域：**

- 这个项目解决什么现实问题？
- 目标用户是谁？他们的痛点是什么？
- 如果这个项目不存在会怎样？（这能帮助理清项目的价值所在）

**技术方案：**

- 核心技术路径是什么？（例如："基于事件驱动的管道，监听链上交易并触发通知"）
- 关键技术选型是什么，为什么这样选？（不要只罗列技术栈，要解释背后的逻辑）
- 数据是如何在系统中流动的？信息如何进入、转换、输出？

**架构：**

- 模块之间的边界在哪里？
- 模块之间如何通信？（HTTP、消息队列、共享状态、函数调用等）
- 外部依赖和集成点有哪些？

**难点与风险：**

- 这个项目解决的最难的技术问题是什么？
- 代码中最复杂或最脆弱的部分在哪里？
- 做了哪些技术权衡，原因是什么？

**模块识别 — 将代码库拆分为 3–8 个逻辑模块：**

- 每个模块应当**内聚**（模块内文件紧密相关）、**松耦合**（通过清晰接口交互）、**以功能命名**（如"交易引擎"、"通知系统"、"用户认证"，而不是"src/lib"这种目录名）
- 对每个模块记录：用途（一句话）、关键文件、复杂度估计（简单 / 中等 / 复杂）

## 写作规范

### 语言

- 全程使用中文撰写
- 使用精确的技术术语，但在首次出现时给出解释
- 每句话都应承载信息，不写废话

### Mermaid 图表 — 必选项，不是可选项

每份 `main.md` 至少包含以下三种图表：

1. **架构总览图**（graph TB/LR）
2. **数据流图**（sequenceDiagram 或 flowchart）
3. **模块依赖图**（graph LR）

Mermaid 语法注意事项：

- 包含中文、空格或特殊字符的标签必须加引号：`A["标签内容"]`
- `subgraph` 嵌套不超过 2 层
- 每个 `subgraph` 必须有对应的 `end`
- 用 `-->` 表示连线，不要用 `---`
- 写完后在脑中验证语法是否正确再输出

### 深度校准

- 小型项目（<20 个文件）：尽量覆盖每个文件
- 中型项目（20–100 个文件）：覆盖所有模块，重点标注关键文件
- 大型项目（100+ 个文件）：聚焦架构和关键流程，以文件路径引用为主

### 阅读路径设计要求

「推荐阅读路径」一节是模块表的导览，**不是清单的复述**。生成时必须满足：

1. **起点选择有理由，且不是依赖最少的模块** —— 起点应选最能体现项目差异化、最能回答"项目凭什么不是另一个 X"的模块；通常是**业务/产品魂模块**而非基础设施模块。三条理由要彼此独立（角色定位 / 读起来的特征 / 作为地图的价值）。
2. **递进路径必须以"上一模块抛出的问题"作为跳转动因** —— 每个 `↓` 后面跟的是一句具体问题（带引号），不是"接下来看 X"这种顺序词。**禁止按目录字母序、文件名序、复杂度序排路径**。如果两个模块互为依赖、应配对阅读，用 `← 当作一对一读` 标注并说明原因。
3. **三条快捷线必须覆盖三种不同读者画像** —— 推荐组合：(a) 想理解产品创新（好奇用户）；(b) 想做二次开发（贡献者）；(c) 想做架构借鉴（学习者）。每条快捷线给出**精确的模块编号 + 具体收益描述**，不能写"读核心模块"这种空话。
4. **必须有"不建议先读"** —— 至少标出 1 个模块作为"延后读"，并说明原因（通常是 UI 壳、产物、纯工具模块）。这条信息对读者价值很高——它防止他们在错误的地方耗费精力。

### 诚实原则

- 如果某段代码令人困惑或文档缺失，直接说明
- 如果项目存在明显问题（安全、性能、架构），在"⚠️ 值得注意的问题"章节中指出
- 如果不确定某个模块的用途，使用"根据代码结构推测，这个模块的作用可能是..."而不是当作事实陈述

## 第四阶段：git 历史演进分析

代码的当前状态只是终点切片。一个仓库当前精巧的架构可能是经过三次推翻重来才形成的——那三次失败和领悟比当前的"正确答案"更有教育意义。这一阶段通过系统性地"考古" git 历史，还原作者的认知与技术演进过程，产出 `.code-insight/evolution.md`。

### 4.1 这一阶段试图回答什么

- 作者最初是怎么理解这个问题的？后来发生了哪些转变？
- 哪些技术/架构选择是一开始就想清楚了的，哪些是被现实倒逼出来的？
- 作者在哪些地方踩过坑、走过弯路？这些教训沉淀成了什么？
- 每一次重要演进，是**内因驱动**（作者经验积累、复杂度自然增长），还是**外因驱动**（生态变化、业务变化、依赖大版本变更），抑或**两者共振**？

### 4.2 首次分析 vs 增量分析

执行本阶段前，**先检查 `.code-insight/evolution.md` 是否已存在**：

- **不存在 → 首次分析**：分析范围是 `git log --reverse`（所有提交），按 4.3 的两遍扫描走全量流程。完成后用 `git rev-parse HEAD` 取当前 HEAD 的完整 hash，写入文档顶部 metadata 的「分析锚点」字段，并在文末「分析锚点与续接信息」表中记录第 1 行。

- **已存在 → 增量分析**：从既有 evolution.md 的「分析锚点与续接信息」表中读取最后一行的 `commit hash`，记为 `LAST_ANALYZED`。
  - 用 `git log --reverse <LAST_ANALYZED>..HEAD` 拉取自上次以来的新提交
  - 若无新提交：告诉用户"自 `<short_hash>` 以来无新提交，无需更新"，跳过本阶段
  - 若有新提交：仅对新增提交执行两遍扫描，并以**增量更新**的方式改写 evolution.md（详见 4.9 模板末的对接说明）
  - 增量更新的两个原则：(a) 既有结论除非被新证据推翻，否则保留；(b) 推翻时要在被改写段落加注更新日期
  - 完成后取新的 HEAD hash，更新 metadata 顶部锚点 + 在续接表追加新行

这样设计的目的：让 evolution.md 成为可持续生长的考古日志，而不是每次重跑都全量重写。

### 4.3 两遍扫描法

直接对所有提交读 diff 既昂贵又会被噪声淹没。采用两遍扫描：

**第一遍：commit message 全量过滤（廉价）**

按时间正序读取所有 commit message，对每条做粗分类：

- ❌ **噪声**：拼写修复、依赖小升级、纯格式化、无冲突的 merge
- ⚠️ **候选**：常规 feature、普通 bug fix
- ✅ **关键**：满足以下任一信号
  - **关键词**：refactor / rewrite / redesign / migrate / drop / introduce / 重构 / 重写 / 迁移 / 替换 / 引入 / 改造 / 重新设计
  - **首次引入新概念**：第一次出现 test / CI / docker / auth / cache / log 等基础设施
  - **大改动**：后续 stat 显示 diff > 200 行 或 > 10 个文件
  - **文件移动/重命名**：架构边界调整的强信号
  - **撤销/还原**：revert 提交，反映方向调整
  - **重大命名变化**：API 重命名往往代表心智模型升级

**第二遍：关键提交深度阅读**

对所有 ✅ 提交，依次：

1. `git show <hash>` 读取完整 diff
2. 对照前后状态，提炼"作者在这个时刻意识到了什么"
3. 标注信号类型：
   - 🧠 **心智模型升级**（重新理解问题本身）
   - 🏗️ **架构调整**（重新组织代码边界）
   - 🔧 **技术栈变更**（替换工具/库/语言特性）
   - 📐 **工程化提升**（新增测试/CI/observability/lint）
   - 💼 **业务理解深化**（新增/删除特性反映对需求的重新认识）

### 4.4 常用命令

```bash
git log --reverse --pretty=format:"%h|%ad|%an|%s" --date=short   # 时间正序的全部提交
git shortlog -s -n                                                # 贡献者分布
git show --stat <hash>                                            # 单提交的文件级统计
git show <hash>                                                   # 单提交的完整 diff
git log --diff-filter=R --name-status                             # 重命名追踪
git log --grep="refactor\|重构\|rewrite\|migrate" -i              # 关键词检索
git log --follow <file>                                           # 单文件演化追踪
```

### 4.5 分析维度

**软实力演进 — 看不见但更深的变化**

- **问题理解的深化**：作者最初把问题定义为 X，后来发现实际是 Y。例如从"做一个爬虫"到"做一个反爬规避策略库"。
- **抽象能力的成熟**：从面向流程 → 出现接口/协议抽象 → 形成领域模型。
- **取舍判断的精进**：从"什么都想做"到大胆删除特性、明确边界。
- **工程哲学的形成**：注释密度、命名风格、目录组织、commit message 文风的变化都是侧写。

**硬实力演进 — 可量化、可追溯**

- **技术栈迭代**：哪些库被引入又被替换？理由（性能/社区/API 设计/维护成本）。
- **架构成型**：单文件 → 分层 → 模块化 → 服务化 的路径与诱因。
- **错误处理与可观测性**：从无 → print → log → 结构化 log → metrics → tracing。
- **测试策略**：是否、何时、如何引入测试？从单元 → 集成 → e2e 的粒度演变。
- **CI/CD 与协作工程**：何时引入构建工具、lint、format、CI、release 流程。

### 4.6 「演进驱动力」的判断框架

每识别出一次演进，必须追问根因：

- **内因**：作者经验积累、项目复杂度自然增长、前一次决策的债务到期
- **外因**：底层依赖大版本变化（如 Python 3.10 → 3.12、React 17 → 18）、行业范式变化（如轮询 → 事件驱动成为主流）、业务方需求转向、安全 CVE 倒逼
- **共振**：通常是内外因同时作用，需说明哪个是主因

判断依据来自：
- 时间点是否吻合外部事件（新版本发布、行业热点）
- diff 中是否出现明确的"被动应对"痕迹（如 `# breaking change in vX.Y`）
- 同期是否有相关依赖更新（看 `package.json` / `requirements.txt` / `go.mod` 的变化）

如果只能观察到现象、看不出根因，**明确标注"根因不可考"，不要硬编故事**。

### 4.7 大型仓库的取舍

| 提交规模 | 策略 |
| -------- | ---- |
| < 50 | 全量分析每个提交 |
| 50–500 | 第一遍全量扫 message，第二遍仅深读 ✅ 关键提交（通常 10–30 个） |
| 500–2000 | 改为**主题式纵向追踪**：先定 4–6 个主题（如"测试演进"、"错误处理演进"），每主题挑 5–10 个代表性提交 |
| 2000+ | 按年/季度分桶抽样，每桶选最具代表性的 3–5 个提交 |
| 仅 1–3 个提交 | 跳过本阶段，在 evolution.md 中说明"项目尚无演进可分析" |

### 4.8 多作者处理

- 用 `git shortlog -s -n` 看贡献分布
- **主要贡献者**（占 70%+ 提交）：作为分析主体
- **次要贡献者**：标注其加入时间、贡献的关键节点和与主作者的风格差异
- **协作模式本身**就是演进观察点（单人 → 多人 → 分工明确 → 引入 review 流程）

### 4.10 写作深度的硬约束（防止泛泛而谈）

evolution.md 失败的最常见模式是停留在概括层面——"项目从 A 切换到 B，因为 A 有问题与愿景冲突"。这样的描述对读者**毫无价值**，因为它没回答任何"为什么"。每个拐点、每条软/硬实力主线写完后，必须按以下检查项自查，**任何一条不通过就重写**。

#### 五条硬性检查

1. **API/技术名是否精确到具名实体？**
   不许只写"用 mcp 协议"，要写"用 stdio JSON-RPC 协议跑 @modelcontextprotocol/sdk"。
   不许只写"换成扩展"，要写"基于 chrome.debugger.attach({tabId}) 的 MV3 service worker 扩展"。

2. **痛点是否落到具体场景或可量化数字？**
   不许写"性能开销"，要写"单次工具调用要走 stdio→Node→CDP 4 跳，延迟 50–100ms"。
   不许写"指纹问题"，要写"playwright evaluate 上下文残留 `__playwright` 全局，被某反爬规则识别"。

3. **新方案的工作原理是否解释到能让读者复现的程度？**
   光给名字（"用 chrome.debugger"）不够，要说出关键 API 的语义和它**为什么**能解决老问题（如：chrome.debugger 是浏览器原生 API，attach 后直连 CDP，跳过中间桥接，调用降至个位数 ms）。

4. **是否引用了具体代码位置？**
   每个拐点至少一处 `file:line` 或 commit hash 的具体引用，让读者可验证。

5. **是否披露了新方案的代价？**
   换技术必有取舍。"无代价"通常等于"作者还没踩到坑"。如果确实未观察到，要明说「目前未观察到明显代价，但需关注 X」。

#### 反模式 vs 合格示例

> ❌ **不合格**：
> "项目从 playwright-mcp 切换到自研 daemon + Chrome 扩展。这是项目身份的换芯——如果不是这一笔，今天的 OpenCLI 就只是又一个 Playwright wrapper；正是因为这一步，'复用浏览器登录态'才成为可能。驱动力判断：内因为主——作者意识到 mcp 协议层的开销和指纹问题与产品愿景冲突；同时受外因催化（MV3 service worker + chrome.debugger API 在 2024-2025 已稳定可用）。"
>
> 问题：通篇没有具体技术细节。"开销"是多少？"指纹"是哪个？"chrome.debugger API"做什么？"自研 daemon"是什么架构？读者读完仍然不懂这个切换在技术层面到底发生了什么。

> ✅ **合格**：
> "项目从 `@playwright/mcp` 切换到基于 `chrome.debugger` API 的 MV3 扩展（commit `b2fa7da`，新增 `src/extension/` 约 200 行）。
>
> **playwright-mcp 的具体问题**：
> 1. 必须由 playwright 拉起独立 Chromium 实例（参见 playwright `chromium.launch()` 默认行为），而用户的 cookies/sessionStorage 在系统 Chrome 中，无法直接复用——这与"复用真实登录态"的产品定位根本冲突。
> 2. 单次工具调用链路：MCP client → stdio JSON-RPC → playwright Node 进程 → CDP WebSocket → 浏览器，4 跳序列化，实测延迟 ~80ms（README 早期 benchmark 章节，commit `a1c3...` 已删）。
> 3. playwright 注入 evaluate 上下文时残留 `window.__playwright_*` 全局符号，被一些反爬规则直接命中。
>
> **新方案如何解决**：`chrome.debugger.attach({tabId})` 是 Chrome 扩展原生 API，扩展 service worker 直接 attach 到用户已登录的 Tab，与 CDP 直连无中间层；调用链路降至 1 跳（service worker → CDP），延迟 ~5ms（src/extension/background.ts:42 的 `sendCommand` 实现）。指纹问题同时消失——浏览器原生 attach 不注入任何上下文标记。
>
> **代价**：(1) 仅支持 Chrome/Edge（Firefox 无 chrome.debugger）；(2) 用户需手动安装 unpacked 扩展，破坏 npx 一键体验；(3) MV3 service worker 30s idle 后会被回收，需要 keep-alive 机制（src/extension/keepalive.ts）。
>
> **代码证据**：核心 attach 在 [src/extension/background.ts:42](...)，原 `src/lib/mcp-client.ts` 在同次提交中删除（diff 净减少 ~340 行）。
>
> **驱动力判断**：内因为主——作者在 commit `9be8f2` 注释中已写'mcp 这层是性能瓶颈'，说明痛点先于决策清晰；外因催化——chrome.debugger 在 MV3 下的稳定可用是 Chrome 88+（2021）就具备的能力，作者并非被新 API 推动，而是产品定位想清楚后回头选了一个早就成熟的工具。"

#### 软实力主线也适用

软实力（"问题理解的深化"）同样不能停留在"作者认识更深刻了"。要落到：
- 哪个 commit 的哪段代码体现了旧理解？（贴片段）
- 哪个 commit 的哪段代码体现了新理解？（贴片段）
- 中间发生了什么具体事件让作者改变？（如某个 issue、某次 revert、某次大量删除）

### 4.11 诚实原则

- **区分事实与推测**：diff 显示的是事实，"作者当时在想什么"是推测，必须明确标注
- **拒绝过度叙事**：不要把零散事件硬编成"英雄成长故事"，承认很多变化没有清晰动机
- **承认盲区**：决策性讨论、口头共识、被否决的方案都不可见
- **不要心理画像**：避免对作者本人作判断（"这个作者很草率/严谨"），只描述可观察的代码痕迹

## 第五阶段：展示与引导

生成 `main.md` 和 `evolution.md` 后：

1. 告诉用户这个项目做什么（一句话概括）
2. 识别出了多少个模块
3. 简述演进考古的最大发现（1–2 个最显著的拐点或转变）
