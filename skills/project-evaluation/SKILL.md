# 项目使用者视角评估

从一个**想要使用这个项目的人**的角度出发，全面评估项目的使用价值、上手成本、生态位和竞争力。产出物放在项目根目录下的 `.project-eval/` 目录中，与代码洞察（`.code-insight/`）完全独立。

## 核心定位

这个技能回答的核心问题是：**"我该不该用这个项目？"**

不同于 codebase-overview（回答"这个项目怎么实现的"），此技能回答：

- 这个项目能帮我解决什么问题？
- 上手难不难？
- 有没有更好的替代品？
- 综合来看值多少分？

---

## 第一阶段：信息收集

### 项目本身

阅读以下内容（按优先级排序）：

1. **README.md** — 项目的门面，反映维护者如何向用户介绍自己
2. **文档目录**（docs/、wiki、官方网站链接）— 使用文档的完整度和质量
3. **package.json / Cargo.toml / pyproject.toml 等** — 依赖数量、版本号、许可证
4. **CHANGELOG.md / Releases** — 更新频率、版本演进方向
5. **Issues 和 PR**（如果在本地有 .git 信息）— 社区活跃度线索
6. **示例代码**（examples/、demo/）— 上手友好度
7. **配置文件** — 使用复杂度的直接体现

如果 `.code-insight/main.md` 已存在，直接引用其中的项目概括和模块信息，避免重复分析。

### 竞品调研

使用 web search 搜索竞品信息。搜索策略：

1. 搜索 "[项目解决的问题] + alternatives"、"[项目名] vs"、"best [问题领域] tools [当前年份]"
2. 搜索 "[项目名] review"、"[项目名] comparison"
3. 查看项目 README 中是否提到了竞品或对比
4. 识别 3–6 个主要竞品（不多于 6 个，聚焦最有代表性的）

对每个竞品，收集：名称、一句话定位、Star 数（如开源）、最近更新时间、核心差异点。

---

## 第二阶段：多维度评估

围绕以下维度进行评估，每个维度独立打分（满分 100）：

### 评估维度说明

| 维度               | 权重建议 | 评估要点                                                                             |
| ------------------ | -------- | ------------------------------------------------------------------------------------ |
| **解决问题的能力** | 25%      | 是否真正解决了目标问题？解决得彻底吗？有没有半成品感？                               |
| **上手成本**       | 20%      | 安装步骤多不多？配置复杂吗？文档清楚吗？有没有好的示例？新手能在 30 分钟内跑起来吗？ |
| **功能完整度**     | 15%      | 核心功能是否齐全？边界场景考虑了吗？API 设计合理吗？                                 |
| **社区与生态**     | 15%      | 社区活跃度如何？Issue 响应速度？有没有插件/扩展生态？Stack Overflow 上能搜到答案吗？ |
| **维护状态**       | 15%      | 最近一次更新是什么时候？更新频率如何？是否有明确的路线图？是否有公司或组织支撑？     |
| **文档质量**       | 10%      | 文档是否完整？是否有 API 参考？是否有教程和示例？文档是否与代码同步？                |

权重是建议值，agent 可以根据项目类型调整。例如对于安全相关的库，可以加入"安全性"维度。

### 打分标准

- **90–100**: 业界标杆，几乎无可挑剔
- **75–89**: 优秀，有明确优势，小瑕疵不影响使用
- **60–74**: 合格，能用但有明显不足
- **40–59**: 勉强，需要额外付出才能用好
- **0–39**: 不推荐，存在严重问题

打分必须给出具体理由，不能只写数字。理由应引用具体事实（如"README 只有 3 行"、"最近一次提交是 2 年前"）。

## 第 2.5 阶段：配置文件深度解读（必做）

### 触发条件

在第一阶段信息收集中，检测项目是否包含配置文件。**如果项目包含任何用户需要编辑的配置文件**（如 `config.yaml`、`.env.example`、`settings.json`、`appsettings.json`、`*.toml`、`*.ini`、`docker-compose.yml` 中的环境变量等），则**必须**执行本阶段，并生成 `.project-eval/config-guide.md`。

判断标准：配置文件是**用户在使用项目时需要阅读或修改的文件**，而非纯开发配置（如 `.eslintrc`、`tsconfig.json`）。如果拿不准，宁可生成也不要跳过。

如果项目确实没有用户配置文件（如纯库项目只需 `import` 即可使用），跳过此阶段。

### 工作方法

对项目中**每一个用户配置文件**，执行以下步骤：

#### 步骤 1：通读配置文件

完整阅读配置文件内容，记录每一个配置项的名称和文件中的注释说明。

#### 步骤 2：追踪源码，理解真实行为

**这是关键步骤。** 仅凭配置文件的注释不足以写出好的指南——注释经常过于简略、有歧义甚至过时。你必须在源码中追踪每个配置项的实际消费方式：

1. **找到配置加载代码**：搜索项目中负责解析配置文件的模块（如 `config.py`、`loader.py`、`settings.ts`），理解每个配置项被读取后映射到了什么内部变量
2. **追踪消费代码**：找到内部变量被使用的位置，理解它实际控制的行为。重点关注：
   - 默认值是什么（写在配置文件中的 vs 写在代码中的可能不同）
   - 是否有环境变量可以覆盖
   - 是否有非显而易见的验证或转换逻辑（如类型转换、范围限制、自动回退）
   - 与其他配置项的联动关系（如 A 开启后 B 才生效）
   - 边界情况（如设为 0 或空字符串时的行为）
3. **识别"隐形配置"**：源码中存在但配置文件中未暴露的可配置项，或者只能通过环境变量设置的配置

#### 步骤 3：撰写解读

对每个配置项，产出以下内容：

- **作用**：一句话说清做什么（比配置文件注释更清晰）
- **默认值**：来自源码的实际默认值
- **建议值**：针对不同使用场景的推荐配置
- **深入说明**（仅对非显而易见的配置项）：结合源码解释它到底怎么工作的。用"如果你设了 X，程序会做 Y"的因果句式，避免堆砌源码细节

### 产出物模板 — `.project-eval/config-guide.md`

文档结构如下（用自然语言描述，非代码块）：

**标题**: `# [项目名称] 配置完全指南`

**头部元信息**: 包含版本号、对应配置文件路径列表

**正文结构**: 按配置文件的段落/分组组织，每个段落包含：

1. 该段落的 YAML/JSON 配置原文片段（方便读者对照）
2. 速查表格：配置项 | 作用 | 默认值 | 建议
3. 对非显而易见的配置项，用独立小节展开解释（标题为配置项名称）

**尾部**: 如果项目支持环境变量覆盖，附一张「环境变量速查表」，列出配置文件字段、对应环境变量名、优先级说明

### 写作要求

- **面向使用者，不是开发者**：解释"这个配置控制什么行为"，而不是"这个配置在代码的第 N 行被读取"。源码引用只在需要消除歧义时出现
- **每个配置项都要覆盖**：即使看起来很简单（如 `debug: false`），也要有一行说明。用户不应该需要去翻源码才能理解任何一个配置项
- **重点解释"不太容易理解的配置"**：识别标准——如果一个配置项满足以下任意条件，就需要展开说明：
  - 配置文件注释不足以理解其完整行为
  - 它与其他配置项有隐式联动
  - 不同取值会导致显著不同的行为分支
  - 设错了可能导致功能异常但不报错
  - 名称容易产生误解（如 `mode` 有多个可选值，各值含义差异大）
- **给出实用建议**：不只是解释"是什么"，还要说"建议怎么设"。可以针对不同用户画像（新手/进阶/特定场景）给出不同建议

## 第三阶段：生成文档

在项目根目录创建 `.project-eval/` 目录。

### 判断：是否需要分层？

- **大多数项目**：生成单个 `main.md` 即可，不需要分层
- **需要分层的情况**：当项目功能面很广（如一个大型框架），不同使用场景差异极大，或者竞品数量多且每个竞品值得单独详细分析时，可以拆分子文件

分层判断标准：如果 `main.md` 超过 300 行，考虑将"竞品详细分析"或"使用场景详解"拆分成独立文件。

### 主文档模板 — `.project-eval/main.md`

主文档应包含以下章节（按顺序）：

1. **标题与元信息**：项目名称、评估时间、版本号
2. **一句话评价**：最直接的使用建议
3. **这个项目是什么**：非技术人员也能理解的说明
4. **核心使用场景**：2-5 个场景，每个含目标、方案、难度、示例
5. **快速上手**：安装命令、最小可运行示例、常见踩坑点
6. **功能全景**：mindmap 图 + 功能支持状态表格
7. **配置指南**（如果生成了 config-guide.md）：简要说明 + 链接到 config-guide.md
8. **竞品分析**：quadrantChart 图 + 横向对比评分表 + 各竞品简评
9. **优势与不足**：3-5 条优势 + 2-4 条不足
10. **综合评分**：六维度打分表 + 加权总分
11. **最终建议**：推荐场景 + 不推荐场景 + 一句话总结

**重要提示：** 生成实际文档时，代码围栏必须使用标准的三个连续反引号，否则无法正常渲染。

### 分层结构（仅在需要时使用）

当内容过长需要拆分时，目录结构如下：

```
.project-eval/
├── main.md                    # 主文档（包含总览、评分、结论）
├── config-guide.md            # 配置完全指南（如果项目有配置文件，必须生成）
├── scenarios.md               # 使用场景详解（如果场景多且复杂）
├── competitors/               # 竞品详细分析目录
│   ├── main.md                # 竞品总览与对比表
│   ├── [竞品a].md             # 竞品A详细分析
│   └── [竞品b].md             # 竞品B详细分析
└── getting-started.md         # 详细上手指南（如果上手流程复杂）
```

拆分规则：

- `main.md` 始终是入口，包含所有关键结论和评分
- `config-guide.md` 是独立文件，在 `main.md` 中通过"配置指南"章节链接引用
- 子文件只展开细节，不包含独立结论
- 子文件必须在 `main.md` 中有链接引用
- 不要超过两层嵌套

## 写作规范

### 立场

- 站在**使用者**的角度，不是开发者或代码审查者的角度
- 关心的是"好不好用"、"值不值得用"，不是"代码写得好不好"
- 语言直白，结论明确，不模棱两可

### 打分纪律

- 不打人情分。如果项目确实不行，该给低分就给低分
- 每一分都要有事实支撑，不允许"感觉还行所以给 70 分"
- 竞品打分必须用同一套标准，不能对当前项目特别宽容
- 如果某个维度因信息不足无法准确评估，标注为"信息不足，暂估 XX 分"并说明原因

### 竞品调研诚实原则

- 如果搜不到足够的竞品信息，诚实说明，不要编造
- 如果某个竞品明显优于当前项目，不要回避这个结论
- 如果当前项目就是细分领域最好的，也不用刻意找缺点来"平衡"

### 图表

每份评估至少包含：

1. **功能全景图**（mindmap）— 让用户一眼看到项目能做什么
2. **竞品定位图**（quadrantChart 或其他对比图）— 直观展示竞争格局

Mermaid 语法规则：

- 包含中文、空格或特殊字符的标签必须加引号：`A["标签"]`
- mindmap 中的根节点用双括号：`root(("名称"))`
- quadrantChart 中的坐标值范围是 0–1 的小数

### 语言

- 全程使用中文撰写
- 面向的是想要使用这个项目的人，不是想要贡献代码的开发者
- 避免深入源码实现细节，聚焦使用体验和外部可观察的行为

---

## 第四阶段：展示与引导

生成文档后：

1. 给出一句话评价
2. 报告综合得分
3. 告知竞品中得分最高的是哪个
4. 如果生成了 `config-guide.md`，告知用户配置指南已生成，并提示"有哪个配置项想进一步了解吗？"
5. 问用户："需要深入了解某个竞品，还是看看具体的使用场景？"

### 后续修正

评估文档是活文档。用户后续提出的疑问、纠错和补充要求，
由 insight-refine 技能自动处理——回答问题的同时将内容写回 md 文件。
